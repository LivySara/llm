# 思考

## 沟通问题：bk-bscp 接口改造项目

### 项目背景
- **任务**：bk-bscp 项目大部分接口 URL 需要改造，添加 `projectId`/`envId` 参数
- **挑战**：个人不熟悉业务流程，改造涉及文件数量庞大

### 遇到的问题

**1. 信息不对称**
- 后端未明确告知哪些接口已改造
- 纯粹要求进行测试和比对（依赖低性能大模型，结果不可靠）

**2. 工作效率低**
- 涉及改造的文件数量过多
- 未能按模块分批处理清楚
- 后期返工了部分工作

**3. 沟通障碍**
- 后端交流不积极
- 改动内容不主动同步
- 需要自己逐一询问确认

### 改进建议

**对后端**：
- 提供改造接口的完整清单
- 明确改动范围和优先级
- 主动同步进度和问题

**对自身**：
- 提前梳理业务流程
- 主动确认改造范围
- 建立清晰的模块划分和检查机制

## AI 使用问题：低性能模型的局限性

### 问题描述
由于预算有限，使用低性能大模型进行编码，导致：
- 代码生成结果可能不准确
- 需要额外花费时间进行 Code Review
- 整体效率反而降低

### 成本分析

**直接成本**：
- 低性能模型 API 费用较低
- 但返工和 review 时间成本增加

**隐性成本**：
- 错误代码调试时间
- 代码质量不可控风险
- 长期维护成本增加

### 优化策略

**1. 混合使用策略**
- 简单任务：使用低性能模型
- 复杂任务：使用高性能模型
- 核心业务：人工编写 + 高性能模型辅助

**2. 质量保障机制**
- 所有 AI 生成代码必须经过 Code Review
- 建立自动化测试验证
- 关键代码人工复核

**3. 提示词优化**
- 提供更详细的上下文
- 明确代码规范和约束
- 要求模型输出思考过程

### 决策建议

| 场景 | 推荐方案 |
|------|----------|
| 简单 CRUD | 低性能模型 + Review |
| 业务逻辑复杂 | 高性能模型 |
| 核心功能 | 人工编写 |
| 紧急需求 | 高性能模型 + Review |

### 总结
低性能模型适合快速原型和简单任务，但对于生产代码，需要权衡成本和质量，在关键场景使用高性能模型或人工编写更为经济。
